# 最高原则

AI 必须始终遵守以下原则（优先级从高到低），所有原则均适用于 SoulCoreKit 全栈框架开发。

---

## 一、架构层原则

### 第一原则：架构优先

**核心精神**：任何代码修改都不得破坏既有架构。

**规则**：

| 推荐做法 | 禁止做法 |
|---------|---------|
| 通过接口抽象实现解耦（如 `IGlassEffect`） | 直接依赖具体实现类 |
| 遵循现有模块划分（core/base/ui/network 等） | 随意跨模块调用或移动文件 |
| 新增功能通过扩展而非修改公共接口 | 修改已有公共接口签名 |
| 保持依赖关系单向流动 | 引入循环依赖 |

**违反后果**：降低模块化程度、增加耦合、影响后续维护和扩展。

**检查方法**：查看新增代码是否遵循现有架构模式，是否引入新的跨层依赖。

**SoulCoreKit 实践**：
- 主题系统通过 `Theme` 单例 + `Style` 设计令牌实现全局统一
- 组件通过 `soul_ui.h` 统一入口导出，避免散列引用

---

### 第二原则：组件优先

**核心精神**：所有 UI 必须优先复用已有组件，严禁重复造轮子。

**规则**：

| 推荐做法 | 禁止做法 |
|---------|---------|
| 优先使用 `soul/ui/soul_ui.h` 中的标准组件 | 在多个页面复制相同 UI 代码 |
| 将使用两次及以上的 UI 抽象为公共组件 | 为单个页面创建专用 UI 代码 |
| 通过继承或组合扩展组件能力 | 修改组件源码以适应特定场景 |
| 向公共组件库贡献改进 | 在业务代码中实现类似功能 |

**违反后果**：代码冗余、维护成本增加、视觉不一致。

**检查方法**：搜索代码库中是否存在重复的 UI 实现模式。

**SoulCoreKit 实践**：
- 所有基础组件（Button/Card/Input 等）统一在 `soul/ui/` 目录下维护
- 毛玻璃效果通过 `IGlassEffect` 接口标准化，`GlassWidget`、`Card`、`SideBar` 等组件复用该机制

---

### 第三原则：职责单一（Single Responsibility Principle）

**核心精神**：每个类只负责一件事情，严禁职责混杂。

**规则**：

| 类 | 职责 | 禁止承担 |
|---|------|---------|
| `HomePage` | 页面布局 | 业务逻辑、网络请求 |
| `MusicCard` | 歌曲卡片展示 | 数据获取、状态管理 |
| `SearchBar` | 搜索输入 UI | 搜索业务逻辑 |
| `PlayerController` | 播放控制 UI | 媒体解码、网络播放 |
| `ThemeManager` | 主题管理 | UI 渲染、业务逻辑 |
| `Animation` | 动画效果 | 业务逻辑、UI 布局 |
| `IconManager` | 图标管理 | 业务逻辑、网络请求 |

**违反后果**：类膨胀、难以测试、难以维护、耦合度高。

**检查方法**：审查类的方法和成员变量，确认是否存在超出单一职责的内容。

---

## 二、设计层原则

### 第四原则：统一设计语言

**核心精神**：整个 UI 组件库必须保持一致的视觉和交互体验。

**设计令牌规范**：

| 设计维度 | 统一方式 | 对应类 |
|---------|---------|--------|
| **颜色** | 通过 `ColorRole` 枚举统一管理 | `Style::color(ColorRole)` |
| **字体** | 通过 `Style::font()` 统一返回 | `Style` |
| **圆角** | 通过 `CornerRadius` 枚举统一管理 | `Style::cornerRadius(CornerRadius)` |
| **间距** | 通过 `Spacing` 枚举统一管理 | `Style::spacing(Spacing)` |
| **阴影** | 通过主题颜色统一生成 | `Theme` |
| **动效** | 通过 `Animation` 静态方法统一应用 | `Animation` |
| **图标** | 通过 `IconManager` 单例统一管理 | `IconManager` |
| **交互方式** | 统一悬停/点击/聚焦反馈模式 | 所有组件一致实现 |

**颜色角色扩展规范**：

| 颜色角色 | 语义 | 浅色主题值 | 深色主题值 |
|---------|------|-----------|-----------|
| `Success` | 成功状态色 | `#10b981` | `#34d399` |
| `GlassBackground` | 毛玻璃背景色 | `#ffffff` | `#1e293b` |
| `GlassTint` | 毛玻璃着色层 | `rgba(28,28,30,65)` | `rgba(255,255,255,40)` |
| `GlowColor` | 发光颜色 | `#6366f1` | `#818cf8` |

**规则**：

| 推荐做法 | 禁止做法 |
|---------|---------|
| 使用 `Theme::instance().style()` 获取设计令牌 | 硬编码颜色值、像素值 |
| 遵循统一的动画模式（呼吸/发光/回弹） | 单个组件使用独立动效模式 |
| 使用标准组件的默认样式 | 为特定页面创建独立设计风格 |
| 通过主题切换实现明暗模式 | 在代码中分支判断颜色 |

**违反后果**：视觉混乱、用户体验不一致、维护难度增加。

**检查方法**：检查代码中是否存在硬编码的颜色、字体、间距等设计值。

---

## 三、实现层原则

### 第五原则：最小改动原则（Minimal Change Principle）

**核心精神**：在满足需求的前提下，优先复用现有架构和组件，禁止过度设计。

**规则**：

| 推荐做法 | 禁止做法 |
|---------|---------|
| 基于现有架构扩展功能 | 因局部功能开发而大规模重构 |
| 复用已有组件和工具类 | 随意引入新的设计模式 |
| 保持公共接口向后兼容 | 修改公共接口导致调用方崩溃 |
| 在现有文件中补充代码 | 随意移动文件或重命名模块 |

**架构调整流程**：

```
分析现有实现 → 说明改动理由 → 评估影响范围 → 提供兼容方案 → 实施改动 → 回归测试
```

**违反后果**：引入不必要的复杂度、增加测试成本、破坏兼容性。

**检查方法**：评估每次修改是否为最小必要改动。

---

### 第六原则：纯虚函数接口规范

**核心精神**：公共组件必须通过纯虚函数接口定义契约，实现类与接口分离。

**规则**：

| 推荐做法 | 禁止做法 |
|---------|---------|
| 定义纯虚函数接口（如 `IGlassEffect`） | 直接使用具体类作为参数类型 |
| 接口方法使用 `= 0` 声明 | 在接口中提供默认实现 |
| 实现类继承接口并实现所有方法 | 跳过实现某些接口方法 |
| 接口与实现分离在不同文件 | 将接口和实现混在同一文件 |

**SoulCoreKit 实践**：
- `IGlassEffect` 定义毛玻璃效果的标准接口
- 所有组件遵循接口优先原则，便于后续替换实现

**违反后果**：难以扩展和替换实现、违反依赖倒置原则。

---

### 第七原则：Qt6 兼容性约束

**核心精神**：代码必须兼容 Qt6，禁止使用已废弃的 API。

**规则**：

| 推荐做法 | 禁止做法 |
|---------|---------|
| `event->globalPosition().toPoint()` | `event->globalPos()` |
| `QPropertyAnimation` 动画属性 | `QWidget::setTransform()` / `setScale()` |
| `QPainterPath` 绘制路径 | 直接操作底层图形 API |
| `QGraphicsBlurEffect` 模糊效果 | 手动像素级模糊 |

**SoulCoreKit 实践**：
- `Window::mousePressEvent()` 已从 `globalPos()` 迁移到 `globalPosition()`
- `Animation` 类使用 `geometry` 属性动画替代 `QTransform`

**违反后果**：编译警告、运行时错误、未来版本兼容性问题。

---

### 第八原则：组件库命名与导出约定

**核心精神**：所有组件必须遵循统一的命名规范和导出约定。

**命名规范**：

| 类型 | 命名规则 | 示例 |
|------|---------|------|
| 类名 | 帕斯卡命名法（PascalCase） | `GlassWidget`、`Progress` |
| 文件名 | 小写蛇形命名法（snake_case） | `glass_widget.h`、`progress.cpp` |
| 枚举 | 帕斯卡命名法 + `Role` 后缀 | `ColorRole`、`CornerRadius` |
| 命名空间 | `sc`（SoulCore 缩写） | `namespace sc` |

**导出约定**：

| 规则 | 说明 |
|------|------|
| 统一入口 | 通过 `soul/ui/soul_ui.h` 导出全部组件 |
| 头文件保护 | 使用 `#ifndef SOUL_UI_XXX_H` 格式 |
| 源文件 | 对应头文件的 `.cpp` 实现 |
| MOC 支持 | 需要信号槽的类必须添加 `Q_OBJECT` 宏 |

**违反后果**：代码混乱、难以查找和维护、编译错误。

---

### 第九原则：测试覆盖要求

**核心精神**：所有公共组件必须有对应的测试覆盖。

**规则**：

| 推荐做法 | 禁止做法 |
|---------|---------|
| 为每个组件编写单元测试 | 跳过组件测试直接上线 |
| 测试组件的核心功能和边界情况 | 仅测试正常路径 |
| 测试主题切换对组件的影响 | 忽略主题兼容性测试 |
| 测试动画效果的正确性 | 跳过动画相关测试 |

**违反后果**：组件缺陷无法及时发现、回归风险增加、用户体验受损。

---

## 原则优先级总结

```
第一原则：架构优先          ← 最高优先级，决定整体结构
    ↓
第二原则：组件优先          ← 决定代码复用策略
    ↓
第三原则：职责单一          ← 决定类设计质量
    ↓
第四原则：统一设计语言      ← 决定视觉一致性
    ↓
第五原则：最小改动原则      ← 决定实现方式
    ↓
第六原则：纯虚函数接口规范  ← 决定接口设计
    ↓
第七原则：Qt6 兼容性约束    ← 决定技术选型
    ↓
第八原则：命名与导出约定    ← 决定代码规范
    ↓
第九原则：测试覆盖要求      ← 决定代码质量保障
```

**说明**：低优先级原则不得违反高优先级原则。当原则之间出现冲突时，优先遵守优先级更高的原则。
