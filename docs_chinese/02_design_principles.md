# SoulCoreKit 设计原则

## 概述

这些原则定义了指导 SoulCoreKit 所有设计和实现决策的基本哲学。它们按优先级排序——低优先级原则不得违反高优先级原则。

---

## 原则一：架构优先

**核心精神**：任何代码变更不得违反现有架构。

**规则**：

| 推荐 | 禁止 |
|------|------|
| 通过接口抽象实现解耦（如 `IGlassEffect`） | 直接依赖具体实现类 |
| 遵循现有模块边界（core/base/ui/network 等） | 任意跨模块调用或文件移动 |
| 通过扩展添加新功能，不修改公共接口 | 修改现有公共接口签名 |
| 保持单向依赖流向 | 引入循环依赖 |

**违反后果**：模块化程度降低，耦合度增加，阻碍未来维护和扩展。

**验证方式**：审查新代码是否遵循现有架构模式，检查是否有新的跨层依赖。

---

## 原则二：组件优先

**核心精神**：所有 UI 必须优先复用现有组件，不得重复造轮子。

**规则**：

| 推荐 | 禁止 |
|------|------|
| 使用 `soul/ui/soul_ui.h` 中的标准组件 | 在多个页面复制相同 UI 代码 |
| 将使用两次及以上的 UI 抽象为公共组件 | 创建页面专属 UI 代码 |
| 通过继承或组合扩展组件能力 | 修改组件源码适配特定场景 |
| 将改进贡献到公共组件库 | 在业务代码中实现类似功能 |

**违反后果**：代码冗余，维护成本增加，视觉不一致。

**验证方式**：搜索代码库中重复的 UI 实现模式。

---

## 原则三：单一职责

**核心精神**：每个类只负责一件事，不混职责。

**类职责矩阵**：

| 类 | 职责 | 禁止 |
|----|------|------|
| `HomePage` | 页面布局 | 业务逻辑、网络请求 |
| `MusicCard` | 歌曲卡片展示 | 数据获取、状态管理 |
| `SearchBar` | 搜索输入 UI | 搜索业务逻辑 |
| `PlayerController` | 播放控制 UI | 媒体解码、网络播放 |
| `ThemeManager` | 主题管理 | UI 渲染、业务逻辑 |
| `Animation` | 动画效果 | 业务逻辑、UI 布局 |
| `IconManager` | 图标管理 | 业务逻辑、网络请求 |

**违反后果**：类膨胀，难以测试，耦合度高。

**验证方式**：审查类的方法和成员变量，识别超出单一用途的职责。

---

## 原则四：统一设计语言

**核心精神**：整个 UI 组件库必须保持一致的视觉和交互体验。

**设计令牌规范**：

| 设计维度 | 统一方式 | 对应类 |
|----------|----------|--------|
| **颜色** | 通过 `ColorRole` 枚举管理 | `Style::color(ColorRole)` |
| **字体** | 通过 `Style::font()` 统一返回 | `Style` |
| **圆角** | 通过 `CornerRadius` 枚举管理 | `Style::cornerRadius(CornerRadius)` |
| **间距** | 通过 `Spacing` 枚举管理 | `Style::spacing(Spacing)` |
| **阴影** | 从主题颜色生成 | `Theme` |
| **动画** | 通过 `Animation` 静态方法应用 | `Animation` |
| **图标** | 通过 `IconManager` 单例管理 | `IconManager` |
| **交互** | 统一的悬停/点击/聚焦反馈模式 | 所有组件 |

**扩展颜色角色**：

| 颜色角色 | 语义 | 亮色主题 | 暗色主题 |
|----------|------|----------|----------|
| `Success` | 成功状态 | `#10b981` | `#34d399` |
| `GlassBackground` | 毛玻璃背景 | `#ffffff` | `#1e293b` |
| `GlassTint` | 毛玻璃着色层 | `rgba(28,28,30,65)` | `rgba(255,255,255,40)` |
| `GlowColor` | 发光效果颜色 | `#6366f1` | `#818cf8` |

**规则**：

| 推荐 | 禁止 |
|------|------|
| 通过 `Theme::instance().style()` 获取设计令牌 | 硬编码颜色值、像素值 |
| 遵循统一动画模式（呼吸/发光/弹跳） | 每个组件使用独立动画模式 |
| 使用标准组件的默认样式 | 为特定页面创建独立设计风格 |
| 通过主题切换实现暗色/亮色模式 | 在代码中按颜色分支 |

**违反后果**：视觉混乱，用户体验不一致，维护难度增加。

**验证方式**：检查代码中是否有硬编码的颜色、字体、间距等设计值。

---

## 原则五：最小变更

**核心精神**：满足需求时优先复用现有架构和组件，不过度设计。

**规则**：

| 推荐 | 禁止 |
|------|------|
| 在现有架构基础上扩展功能 | 为局部功能开发进行大规模重构 |
| 复用现有组件和工具类 | 随意引入新设计模式 |
| 保持公共接口向后兼容 | 修改公共接口导致调用方崩溃 |
| 在现有文件中添加代码 | 随意移动文件或重命名模块 |

**架构调整流程**：

```
分析现有实现 → 说明修改理由 → 评估影响范围 → 提供兼容方案 → 实施变更 → 回归测试
```

**违反后果**：不必要的复杂性，测试成本增加，兼容性破坏。

**验证方式**：评估每项修改是否为最小必要变更。

---

## 原则六：接口优先

**核心精神**：公共组件必须通过纯虚函数接口定义契约，实现与接口分离。

**规则**：

| 推荐 | 禁止 |
|------|------|
| 定义纯虚函数接口（如 `IGlassEffect`） | 直接使用具体类作为参数类型 |
| 接口方法声明为 `= 0` | 在接口中提供默认实现 |
| 实现类继承接口并实现所有方法 | 跳过实现某些接口方法 |
| 接口和实现在不同文件中分离 | 在同一文件中混合接口和实现 |

**SoulCoreKit 实践**：
- `IGlassEffect` 定义了毛玻璃效果的标准接口
- 所有组件遵循接口优先原则，便于未来替换

**违反后果**：难以扩展和替换实现，违反依赖倒置原则。

---

## 原则七：Qt6 兼容性

**核心精神**：代码必须兼容 Qt6，不得使用已弃用的 API。

**规则**：

| 推荐 | 禁止 |
|------|------|
| `event->globalPosition().toPoint()` | `event->globalPos()` |
| 使用 `QPropertyAnimation` 进行属性动画 | `QWidget::setTransform()` / `setScale()` |
| 使用 `QPainterPath` 绘制路径 | 直接操作底层图形 API |
| 使用 `QGraphicsBlurEffect` 实现模糊效果 | 手动像素级模糊 |

**SoulCoreKit 实践**：
- `Window::mousePressEvent()` 从 `globalPos()` 迁移到 `globalPosition()`
- `Animation` 类使用 `geometry` 属性动画而非 `QTransform`

**违反后果**：编译警告，运行时错误，未来版本兼容性问题。

---

## 原则八：命名与导出规范

**核心精神**：所有组件必须遵循统一的命名和导出规范。

**命名规范**：

| 类型 | 规则 | 示例 |
|------|------|------|
| 类 | PascalCase | `GlassWidget`, `Progress` |
| 文件 | snake_case | `glass_widget.h`, `progress.cpp` |
| 枚举 | PascalCase，后缀 `Role` | `ColorRole`, `CornerRadius` |
| 命名空间 | `sc`（SoulCore 缩写） | `namespace sc` |

**导出规范**：

| 规则 | 描述 |
|------|------|
| 统一入口 | 通过 `soul/ui/soul_ui.h` 导出所有组件 |
| 头文件保护 | 使用 `#ifndef SOUL_UI_XXX_H` 格式 |
| 源文件 | 对应头文件的 `.cpp` 实现 |
| MOC 支持 | 需要信号/槽的类必须添加 `Q_OBJECT` 宏 |

**违反后果**：代码混乱，难以查找和维护，编译错误。

---

## 原则九：测试覆盖

**核心精神**：所有公共组件必须有相应的测试覆盖。

**规则**：

| 推荐 | 禁止 |
|------|------|
| 为每个组件编写单元测试 | 发布前跳过组件测试 |
| 测试核心功能和边界情况 | 只测试正常路径 |
| 测试主题切换对组件的影响 | 忽略主题兼容性测试 |
| 测试动画正确性 | 跳过动画相关测试 |

**违反后果**：组件缺陷未被发现，回归风险增加，用户体验受损。

---

## 原则优先级总结

```
原则一：架构优先            ← 最高优先级，决定整体结构
    ↓
原则二：组件优先            ← 决定代码复用策略
    ↓
原则三：单一职责            ← 决定类设计质量
    ↓
原则四：统一设计语言        ← 决定视觉一致性
    ↓
原则五：最小变更            ← 决定实现方式
    ↓
原则六：接口优先            ← 决定接口设计
    ↓
原则七：Qt6 兼容性          ← 决定技术选型
    ↓
原则八：命名与导出          ← 决定代码规范
    ↓
原则九：测试覆盖            ← 决定代码质量保障
```

**注意**：低优先级原则不得违反高优先级原则。当原则之间发生冲突时，高优先级原则优先。
